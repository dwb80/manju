# 六、产品设计原则

这是后续所有功能设计都要遵循的原则。

### 原则一：资产优先（Asset First）

任何可复用内容都应沉淀为资产，例如：

* 角色
* 场景
* Prompt
* 音效
* 配音
* LoRA
* 工作流
* 模板

目标是避免重复创建。

---

### 原则二：AI 原生（AI Native）

AI 不只是一个按钮，而是贯穿整个生产流程。

例如：

```
剧本
    ↓ AI 优化
分镜
    ↓ AI 生成
Prompt
    ↓ AI 优化
图片
    ↓ AI 批量生成
视频
    ↓ AI 生成
审核
    ↓ AI 预审
发布
    ↓ AI 推荐发布时间
```

---

### 原则三：工作流驱动（Workflow Driven）

不是页面驱动，而是流程驱动。

例如：

```
待编剧

↓

待分镜

↓

待图片

↓

待视频

↓

待审核

↓

已发布
```

系统围绕状态推进，而不是让用户自己管理进度。

---

### 原则四：全链路可追溯（Traceability）

任何内容都应能追溯到来源。

例如：

```
视频

↓

由哪些图片生成

↓

图片使用哪个 Prompt

↓

Prompt 使用哪个模板

↓

模板属于哪个项目

↓

由哪个成员修改
```

形成完整的血缘关系（Lineage）。

---

### 原则五：平台化（Platform First）

所有 AI 能力都通过统一接口调用。

例如：

```
AI图片

↓

统一 AI Service

↓

Provider

↓

即梦
Flux
SDXL
ComfyUI
```

业务模块无需关心具体模型。

# 为什么分镜作为核心生产单元？

因为 AI 漫剧和传统影视最大的区别：

传统影视：

```
剧本
 ↓
拍摄
 ↓
剪辑
```

AI漫剧：

```
剧本
 ↓
镜头设计
 ↓
生成画面
 ↓
生成运动
 ↓
合成
```

中间多了一层：

> **分镜（Storyboard）**

它连接：

* 创意
* AI模型
* 图片
* 视频
* 音频
* 审核

所以它必须成为核心对象。

# 5.2 为什么增加 Scene 层？

因为影视生产天然存在两个维度：

## 编剧维度

关注：

> 发生了什么故事？

例如：

```
第一场：
茶信馆退婚

人物：
林逸
萧晓

事件：
双方解除婚约
```

这是 Scene。

---

## 导演/AI生产维度

关注：

> 怎么拍？

例如：

```
镜头001：

远景

茶馆门口

林逸走出


镜头002：

特写

林逸眼神


镜头003：

中景

萧晓出现
```

这是 Shot。

---

因此：

```
Scene = 剧情逻辑单元

Shot = AI生产单元
```

# 11.1 数据库总体设计原则

---

# 一、设计原则

## 1. 项目隔离

所有业务数据必须关联：

```text
project_id
```

例如：

角色：

```
林逸
```

实际：

```
project_id=10001
character_id=50001
```

---

## 2. AI生产可追溯

任何 AI 结果必须能够追踪：

```text
谁生成

什么时候生成

使用什么模型

输入是什么

输出是什么
```

---

## 3. 资产版本化

所有重要资产：

支持：

```text
V1

V2

V3
```

---

## 4. 业务与AI解耦

业务：

```
Shot
```

不要直接绑定：

```
FLUX
```

而绑定：

```
AI Strategy
```

---

结构：

```text
Shot

↓

AI Task

↓

Model Strategy

↓

Model
```
