本文档定义了 AI 漫剧项目管理系统的核心业务对象及其关系，这是数据库设计和系统架构的基础。整个系统围绕以下核心对象构建。

# 2.3 核心业务对象设计

整个系统围绕以下对象：

---

## 1. Project（项目）

### 代表

一部漫剧。

例如：

```
灵契界
```

### 包含

```
项目名称
季度
负责人
状态
成员
```

### 核心属性

- **基本信息**
  - 项目名称：项目的唯一标识名称
  - 项目类型：漫剧、短剧、漫画等类型分类
  - 项目风格：古风、现代、科幻等风格定位
  - 目标受众：用户群体定位
  - 发布平台：抖音、快手、B站等发布渠道

- **组织信息**
  - 负责人：项目主要负责人
  - 管理员：项目管理员列表
  - 成员：项目参与成员列表
  - 团队角色：成员在项目中的角色分配

- **状态信息**
  - 项目状态：筹备、制作中、审核、已发布等
  - 进度百分比：整体完成进度
  - 当前阶段：当前所处的生产阶段
  - 预计完成时间：项目预计完成时间

- **资源配置**
  - 剧集数量：项目包含的剧集数量
  - 资产库：项目专属的资产库
  - 工作流配置：项目的生产工作流配置

## 关系详解

### 项目与剧集

- **关系类型**：一对多（1:N）
- **业务含义**：一个项目包含多个剧集
- **数据约束**：剧集必须属于某个项目

### 剧集与分镜

- **关系类型**：一对多（1:N）
- **业务含义**：一个剧集包含多个分镜
- **数据约束**：分镜必须属于某个剧集

### 项目与角色资产

- **关系类型**：一对多（1:N）
- **业务含义**：一个项目可以创建多个角色资产
- **数据约束**：角色资产可以属于项目或全局资产库

### 项目与场景资产

- **关系类型**：一对多（1:N）
- **业务含义**：一个项目可以创建多个场景资产
- **数据约束**：场景资产可以属于项目或全局资产库

### 分镜与图片

- **关系类型**：一对多（1:N）
- **业务含义**：一个分镜可以生成多个图片版本
- **数据约束**：图片必须关联某个分镜

### 分镜与视频

- **关系类型**：一对多（1:N）
- **业务含义**：一个分镜可以生成多个视频版本
- **数据约束**：视频必须关联某个分镜

### 剧本与分镜

- **关系类型**：一对一或一对多（1:1或1:N）
- **业务含义**：剧本指导分镜制作，一个剧本对应多个分镜
- **数据约束**：分镜需要参考剧本内容

### 工作流与AI任务

- **关系类型**：一对多（1:N）
- **业务含义**：一个工作流执行产生多个AI任务
- **数据约束**：AI任务由工作流节点触发

### AI任务与模型

- **关系类型**：多对一（N:1）
- **业务含义**：多个AI任务使用同一个模型
- **数据约束**：AI任务必须指定使用的模型

## 数据模型设计原则

### 1. 领域驱动设计（DDD）

采用领域驱动设计方法，以业务领域为核心，确保数据模型与业务模型的一致性。

### 2. 资产优先原则

所有可复用内容都设计为资产对象，支持跨项目引用和复用。

### 3. 可追溯性原则

建立完整的对象关系链，支持从最终成品追溯到原材料。

### 4. 版本管理原则

关键对象（剧本、Prompt、资产等）都支持版本管理，记录变更历史。

### 5. 状态管理原则

所有生产对象都有明确的状态定义，支持状态流转和状态查询。

---

## 数据模型扩展性

### 支持扩展的对象类型

- **资产类型扩展**：支持新增Prompt资产、LoRA资产、工作流模板等
- **AI能力扩展**：支持新增音频生成、3D生成等AI能力
- **内容类型扩展**：支持新增小说、漫画、动画等内容类型

### 支持扩展的关系类型

- **跨项目引用**：支持项目间资产引用和共享
- **资产组合**：支持多个资产组合成新资产
- **工作流模板化**：支持工作流模板化和复用
