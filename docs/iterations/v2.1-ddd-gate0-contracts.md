## 2. 状态机冻结

### 2.1 Shot

| 当前状态 | 允许命令 | 目标状态 | 必要条件 | 事件 |
|---|---|---|---|---|
| `draft` | `markReady` | `ready` | 必要元数据齐全 | 无 |
| `draft/ready/needs_fix/rejected` | `startGeneration` | `generating` | 未归档、命令未处理 | 无 |
| `generating` | `attachGeneratedVideo` | `ready` | candidate/provider request 匹配当前生成请求 | `ShotVideoCandidateAttached` |
| `ready/needs_fix` | `submitForReview` | `in_review` | 有有效生成结果 | `ShotSubmittedForReview` |
| `in_review` | `approve` | `approved` | 有有效 `reviewId` | `ShotApproved` |
| `in_review` | `reject` | `rejected` | 有有效 `reviewId` 和原因 | `ShotRejected` |
| `in_review/approved/rejected` | `requestFix` | `needs_fix` | 记录来源与原因 | 无 |
| `ready/approved/needs_fix/rejected` | `archive` | `archived` | 无进行中的生成/审核 | 无 |
| `archived` | `restore` | `draft` | 未软删除 | 无 |

`editMetadata` 不改变状态、版本输入、审核结果或 Provider 结果。`softDelete` 不改变状态，但 `in_review/approved` 禁止普通删除。每个成功命令只增加一次聚合版本。

### 2.2 Review

权威存储是 `review_items`。旧 `reviews` 仅保留只读兼容或迁移，不再维护同一审核事实。

| 当前状态 | 允许命令 | 目标状态 | 必要条件 | 事件 |
|---|---|---|---|---|
| 不存在 | `submit` | `pending` | target/project/submittedBy 有效 | `ReviewSubmitted` |
| `pending` | `start` | `in_review` | reviewer 有效 | 无 |
| `pending` | `cancel` | `cancelled` | 尚未开始审核 | 无 |
| `in_review` | `approve` | `approved` | 同版本未处理 | `ReviewApproved` |
| `in_review` | `reject` | `rejected` | 有有效原因 | `ReviewRejected` |
| `rejected` | `markNeedsFix` | `needs_fix` | 驳回历史已写入 | 无 |
| `needs_fix` | `resubmit` | `pending` | 新内容版本有效 | `ReviewResubmitted` |
| `approved/rejected` | `close` | `closed` | 审核结果已持久化 | 无 |

`closed/cancelled` 是终态。审批和驳回不允许从 `pending` 执行。SLA 服务只能修改 SLA 元数据，不能修改审核状态。历史由聚合产生，调用方不能传入。

### 2.3 PipelineRun 与 PipelineNode

Run：

- `pending → running`：`startRun`
- `running → paused`：`pauseRun`
- `paused → running`：`resumeRun`
- `running → completed | failed`：`finalizeRun`
- `failed → running`：仅显式恢复/重试命令且策略允许
- `completed`：不可逆终态

Node：

- `pending → running | paused | skipped`
- `pending → completed`：仅幂等缓存命中，并仍由聚合决定
- `running → completed | retrying | failed`
- `retrying → running | paused | skipped | failed`
- `paused → pending | skipped`
- `failed → pending`：仅 `retryNode` 且重试策略允许，不清零累计次数
- `completed/skipped`：不可逆终态

Executor 的 `{ kind: "cancelled" }` 本期映射为 `failed`，错误分类为 `cancelled`；本期不新增 Node 状态。Scheduler 在 Run 为 `paused` 时不得选出新节点，并在实际调度前再次检查版本和预算。

DAG 条件类型冻结为 `always | on_approve | on_reject | on_skip`。V2.1 不接受 `expression`；不得静默归一为 `always`。所有入边在源节点终态后都必须被标记为“已求值”，避免混合条件分支卡死。

## 4. 事件与幂等契约

事件名称以 `DOMAIN_EVENT_TYPES` 为准，Outbox `topic = DomainEvent.type`。完整领域 Envelope 放入现有 Outbox payload，格式为：

```text
{
  aggregateId,
  aggregateType,
  occurredAt,
  eventVersion: 1,
  payload
}
```

兼容规则：

- 已发布字段不删除、不改名、不改变含义。
- 新字段只能追加为可选字段；破坏性变更必须发布新事件名或提高 `eventVersion`。
- 事件只携带稳定业务 ID，不携带聚合对象、Repository、Request 或 Provider 对象。
- 消费者使用 `event.id` 派生确定性的 `commandId`，已处理命令返回 `command_already_processed`，不得重复推进状态。
- Review/Shot 事件可携带 `pipelineRunId/pipelineNodeId`；存在 Pipeline 关联时必须携带，避免消费者猜测。
- `ReviewRejected` 和 `ShotRejected` 的 `reason` 必填，由各自聚合的专用 Payload 类型进一步收紧。
- 现有 snake_case `pipeline_events` 仅作为审计/SSE 投影，不作为权威领域事件。
- 旧 `run.lifecycle.created` 暂作为基础设施兼容事件；不冒充本期冻结的领域事件。

## 5. 事务与迁移顺序

聚合 Repository 必须在共享 SQLite 事务内完成状态、Node/历史/快照和 Outbox 写入。禁止先提交状态，再单独 enqueue Outbox，也禁止吞掉 Outbox/历史/快照写入错误。

迁移执行顺序冻结为：

1. `add-shot-aggregate-version`
2. `add-review-aggregate-version`
3. `add-pipeline-run-aggregate-version`

每个迁移必须：

- 将现有 NULL/缺失版本回填为 `1`。
- 将版本约束为非空正整数并提供默认值。
- 可重复执行。
- 在单事务中完成。
- 提供升级、备份恢复和失败回滚测试。

Pipeline Run 和 Node 都需要独立 `version`。聚合保存必须使用 `WHERE id = ? AND version = ?` 并检查影响行数，影响行数为 0 时抛出 `aggregate_version_conflict`。
