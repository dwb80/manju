## 1. 一页说明

### 1.1 为什么做

当前平台已经具备模块化单体、Use Case、Repository、状态机和事件流等 DDD Lite 基础，但三个核心对象的关键状态仍可能被 Service、Scheduler、Executor 或 Router 通过通用 Repository 直接修改。

本迭代要解决的核心问题不是“增加更多目录或 Class”，而是：

1. `PipelineRun`、`Shot`、`Review` 的关键状态只能通过各自聚合行为修改。
2. 非法状态转换必须被领域层拒绝。
3. 并发修改不能静默覆盖，必须通过聚合版本产生明确冲突。
4. 跨聚合协作通过命令和领域事件完成，不允许互相直接改表。
5. 状态、历史、快照和 Outbox 事件必须在同一事务中提交。
6. CI 必须阻止后续代码重新绕过聚合。

### 1.2 怎么分工

三条开发任务并行推进，每条任务负责一个聚合的完整垂直切片：

| 任务线 | 聚合 | 主责范围 | 相对工作量 |
|---|---|---|---:|
| A | `Shot` | 镜头状态、生成候选、送审、审核结果、版本快照 | 中 |
| B | `Review` | 审核状态、审核历史、驳回、重新提交、SLA 边界 | 中高 |
| C | `PipelineRun` | Run/Node 状态、DAG、调度结果、重试、暂停与恢复 | 高 |
| 协调者 | 公共能力 | 契约、共享文件、事件装配、数据库迁移顺序、全量验收 | 持续进行 |

三条任务线不能修改对方聚合的状态，也不能同时修改公共装配文件。

### 1.3 什么时候算完成

只有同时满足以下条件才算完成：

- 三个聚合的所有关键状态转换都有纯领域单元测试。
- 白名单目录之外不存在三个聚合关键状态的直接 Repository 更新。
- 并发写入通过 `version` 乐观锁检测。
- Review → Shot → Pipeline 的跨聚合链路有集成测试。
- 聚合保存、历史/快照和 Outbox 写入具备事务原子性。
- 后端全量门禁、关键 E2E、数据库迁移与恢复测试全部通过。

## 2. 迭代范围

### 2.1 本迭代包含

- 新建纯领域层、应用命令层和三个专属聚合 Repository。
- 将三个对象的状态机和不变量迁入聚合。
- 增加聚合版本与乐观锁。
- 复用并完善现有 Transaction/Outbox 基础。
- 迁移现有直接状态写入口。
- 增加领域事件消费者和跨聚合集成测试。
- 增加禁止绕过聚合的架构门禁。
- 更新相关架构、API 和状态机文档。

### 2.2 本迭代不包含

- 不拆分微服务。
- 不替换 SQLite。
- 不全面重写其他 CRUD 模块。
- 不要求所有 TypeScript 类型都改成 Class。
- 不把普通列表和报表查询强制改为聚合加载。
- 不改变前端产品流程，除非现有接口允许直接提交受保护状态。
- 不顺带重构与三个聚合无关的代码。

## 3. 总体设计约束

### 3.1 依赖方向

```text
HTTP / Event Handler
        ↓
Application Command Handler
        ↓
Domain Aggregate
        ↓
Repository Port
        ↓
SQLite Repository Adapter
```

领域层必须满足：

- 不依赖 `AppContext`。
- 不依赖 HTTP、SQLite、AI Provider 或文件系统。
- 不直接调用另一个聚合的 Repository。
- 可以通过纯内存测试完成状态机和不变量验证。

### 3.2 公共接口

由协调者在三条任务线进入迁移阶段前冻结以下接口：

```ts
export interface AggregateRoot {
  readonly id: string;
  readonly version: number;
  pullDomainEvents(): DomainEvent[];
}

export interface AggregateRepository<TAggregate> {
  get(id: string): Promise<TAggregate | null>;
  save(aggregate: TAggregate, expectedVersion: number): Promise<void>;
}

export interface DomainEvent<TPayload = unknown> {
  id: string;
  type: string;
  aggregateId: string;
  aggregateType: string;
  occurredAt: string;
  payload: TPayload;
}
```

公共错误至少包括：

| 错误码 | 含义 |
|---|---|
| `aggregate_not_found` | 聚合不存在 |
| `invalid_state_transition` | 当前状态不允许执行该行为 |
| `aggregate_version_conflict` | 聚合已被其他请求修改 |
| `aggregate_invariant_violated` | 操作破坏业务不变量 |
| `command_already_processed` | 幂等命令已经处理 |

### 3.3 持久化规则

1. 三个聚合都必须有非空整数 `version`。
2. 保存使用 `WHERE id = ? AND version = ?`。
3. 更新成功后 `version = version + 1`。
4. 影响行数为 0 时返回 `aggregate_version_conflict`。
5. 聚合状态、历史/快照、Outbox 必须在同一事务中提交。
6. 普通查询允许继续使用 Read Model，不要求加载完整聚合。

## 9. 跨聚合协作

### 9.1 Review 驱动 Shot

```text
ReviewApproved
    → ReviewApprovedHandler
    → ApplyShotReviewResultCommand
    → ShotAggregate.approve(reviewId)
```

```text
ReviewRejected
    → ReviewRejectedHandler
    → ApplyShotReviewResultCommand
    → ShotAggregate.reject(reviewId, reason)
```

Review 任务只负责产生审核事件，不直接修改 Shot。

### 9.2 Shot 驱动 Review

```text
ShotSubmittedForReview
    → CreateReviewForShotHandler
    → SubmitReviewCommand
    → ReviewAggregate.submit()
```

Shot 任务只负责表达“镜头已送审”，不直接插入 `review_items`。

### 9.3 Shot/Review 驱动 Pipeline

```text
ShotApproved 或 ReviewApproved
    → PipelineReviewResultHandler
    → CompletePipelineReviewNodeCommand
    → PipelineRunAggregate.completeNode()
```

```text
ReviewRejected
    → PipelineReviewResultHandler
    → Fail/RoutePipelineReviewNodeCommand
    → PipelineRunAggregate 执行返工分支
```

事件消费者和最终装配由协调者负责，避免任一任务线跨边界直接写另一个聚合。

- 领域事件。
- 纯单元测试。

此阶段不迁移现有 API。

### Gate 2：持久化与并发

三条任务线并行完成：

- Mapper。
- 专属 Repository。
- 聚合版本。
- 乐观锁。
- Unit of Work/Outbox 接入。
- Repository 集成测试。

### Gate 3：旧写路径迁移

三条任务线迁移各自负责的 Service。公共 Router、AppContext、Use Case 装配由协调者统一处理。

### Gate 4：跨聚合集成

协调者完成：

- Shot → Review。
- Review → Shot。
- Review/Shot → Pipeline。
- 重试和死信处理。
- 全链路集成测试。

### Gate 5：收紧门禁

- 删除或封闭通用写入口。
- 启用架构扫描。
- 清理兼容代码。
- 全量回归。

---

## 11. 工作量再平衡

PipelineRun 工作量明显高于另外两个聚合。为避免长尾：

1. Shot 负责人完成任务 A 后，支援 Pipeline 的直接写路径迁移和回归测试。
2. Review 负责人完成任务 B 后，支援 Pipeline 的审核、返工和事件链路测试。
3. Pipeline 负责人始终拥有 Pipeline 聚合、状态机和 DAG Policy 的最终修改权。
4. 支援人员不得自行改变 Pipeline 状态机，应通过评审后的命令接口工作。

---

## 12. 提交与评审规则

### 12.1 建议提交粒度

每条任务线按以下顺序提交：

```text
1. domain: 聚合、状态机、领域错误和纯单测
2. persistence: mapper、repository、乐观锁
3. application: commands 和 handlers
4. migration: 旧服务写路径迁移
5. tests: 集成、并发、回归和残留扫描
```

禁止将全部改动压成一个难以评审的大提交。

### 12.2 每日同步内容

每条任务线只需同步以下五项：

```text
已完成：
正在做：
阻塞项：
需要协调者修改的公共文件：
发现的跨聚合风险：
```

### 12.3 评审重点

评审人员优先检查：

- 是否仍能绕过聚合修改关键状态。
- 规则是否在领域层而不是 Router/Service 中重复。
- 事件是否携带了不必要的完整对象。
- 是否正确处理并发版本冲突。
- 是否保证状态、历史、快照和 Outbox 原子性。
- 查询是否被不必要地复杂化。

- [x] 白名单外禁止 `pipelineNodes.update`。

## 15. 风险与处理

| 风险 | 影响 | 处理方式 |
|---|---|---|
| 三条任务线各自实现公共抽象 | 难以合并 | Gate 0 冻结公共契约，公共目录归协调者 |
| `reviews` 与 `review_items` 双模型 | 审核事实不一致 | Review 任务先确定权威模型与兼容策略 |
| Pipeline 聚合过大 | 加载与并发成本上升 | 先以业务一致性边界实现，压测后再决定是否拆 Node 聚合 |
| Scheduler 与人工操作竞态 | 状态被覆盖 | 乐观锁、幂等命令、暂停状态二次检查 |
| 事件重复消费 | 重复推进业务 | 使用 event ID/command ID 去重 |
| 兼容接口继续接受状态 patch | 聚合可被绕过 | DTO 移除受保护字段并增加 CI 门禁 |
| 并发修改共享文件 | 合并冲突 | 公共文件由协调者独占 |
| 重构期间回归范围过大 | 发布风险 | 按 Gate 渐进迁移，每阶段保持测试可运行 |
